## The Product

The stablecoin wallet. Send anywhere. Zero gas. Always.

Your stablecoins live on Magnus. From one balance:
- **Deposit from any chain** — Sign a message. USDT appears. No gas tokens needed. Ever.
- **Send to any chain** — Ethereum, Solana, Tron, BNB, Base. Zero gas fees. Magnus handles it.
- **Send to any bank account** — VietQR, M-Pesa, GCash, UPI, PIX. One transaction, 300ms.
- **Pay only in stablecoins** — Every fee is in USDT/USDC. No ETH, SOL, TRX, or any gas token.

```
  My Wallet (Magnus)
  ┌──────────────────────────────────┐
  │  USDT:      500.00               │
  │  USDC:      200.00               │
  │  VND:   5,000,000                │
  │                                  │
  │  [Deposit]  ← sign message, any chain, gas-free
  │  [Send to chain]  → any chain, zero destination gas
  │  [Send to bank]   → any country, 300ms
  └──────────────────────────────────┘
```

The user never buys gas tokens. Never thinks about which chain their USDT is "on." Never pays ETH to move USDT. Magnus is the single source of truth for your stablecoin balance, and it routes everywhere, gas-free.

## Zero Gas — How It Works

### Depositing to Magnus (any chain → Magnus, gas-free)

The user has USDT on Ethereum but no ETH for gas. Today, that means they're stuck. On Magnus:

```
USER (Ethereum, has USDT, NO ETH)          RELAYER (has ETH)
     │                                          │
     │  1. Signs two off-chain messages          │
     │     (free, no gas, just signatures):      │
     │                                          │
     │     a. Permit: "I approve MagnusBridge    │
     │        to spend 100 USDT"                 │
     │     b. Intent: "Lock 100 USDT, send to    │
     │        my Magnus address"                 │
     │                                          │
     │  ── signed messages ────────────────────→ │
     │                                          │
     │                              2. Relayer submits ONE tx
     │                                 on Ethereum (pays ETH gas)
     │                                          │
     │                                 MagnusBridge.lockWithPermit(
     │                                   USDT, 100,
     │                                   userMagnusAddr,
     │                                   permit, relayerFee: 0.10
     │                                 )
     │                                          │
     │                              3. On Magnus:
     │  ◄── 99.90 USDT on Magnus ──────────────│
     │      (0.10 USDT → relayer as fee)        │
     │                                          │
  User signed a message. USDT appeared. Never had ETH.
```

### Sending from Magnus (Magnus → any chain, gas-free)

```
USER (Magnus, has USDT)
     │
     │  "Send 100 USDT to 0xABC on Ethereum"
     │  Fee: 0.05 USDT (stablecoin, not gas token)
     │
     ▼
  NETTING CHECK:
     Is someone depositing ~100 USDT from Ethereum right now?
     
     YES (90% of the time):
       Net it out. Instant. No bridge traffic.
       User's USDT debited. 0xABC gets USDT from incoming deposit.
       Zero bridge transactions. Zero gas on Ethereum.
     
     NO (10% of the time):
       Magnus validators produce BLS certificate.
       Relayer calls MagnusBridge.release() on Ethereum.
       Relayer pays ETH gas, reimbursed from protocol treasury.
       
  Either way: user paid 0.05 USDT. Touched zero gas tokens.
```

### Internal transfers (Magnus → Magnus, near-free)

```
  Alice sends 100 USDT to Bob on Magnus.
  Fee: ~0.001 USDT. Confirmed in 300ms.
  No bridge. No gas token. Cheapest and fastest path.
  This is why users should HOLD stablecoins on Magnus.
```

### The economics: why gas-free is sustainable

```
  USER PAYS:     0.05 USDT per cross-chain transaction
  
  PROTOCOL COST:
    90% of transfers net out      → $0 bridge cost
    10% need actual bridge        → ~$2 ETH gas each
    Average cost per tx:          → $0.20
    
  At 10,000 daily transactions:
    Revenue: 10,000 × $0.05 =     $500/day
    Cost: 1,000 × $2 =            $2,000/day  ← NOT PROFITABLE YET
    
  At 100,000 daily transactions (95% netting):
    Revenue: 100,000 × $0.05 =    $5,000/day
    Cost: 5,000 × $2 =            $10,000/day ← STILL TIGHT
    
  At 100,000 daily + 0.1% fee on volume ($100 avg):
    Revenue: 100,000 × $0.10 =    $10,000/day
    Cost: 5,000 × $2 =            $10,000/day ← BREAKEVEN
    
  At 1M daily (96% netting, CLS-level):
    Revenue: 1,000,000 × $0.10 =  $100,000/day
    Cost: 40,000 × $2 =           $80,000/day ← PROFITABLE
    
  Gas-free becomes sustainably profitable at scale.
  Before scale: subsidize from treasury / fundraise.
  Same playbook as Uber (subsidized rides → network effect → profit).
```

## Why This Needs to Exist

The #1 UX failure in crypto: you need gas tokens you don't want to hold, just to move money you already have.

A person holding 100 USDT on Ethereum cannot send it to anyone without first buying ETH. A person holding USDT on Tron needs TRX. On Solana, SOL. On BNB, BNB. Across 125 chains, 125 different gas tokens. Each one requires a separate purchase, a separate wallet setup, a separate understanding of fee markets.

For the 66% of stablecoin holders in emerging markets (Goldman Sachs), this is the barrier. They have stablecoins from remittances, freelancing, or local exchange purchases. They want to send money to a bank account or to someone on a different chain. They can't, because they don't have the gas token.

**The stablecoin fragmentation numbers:**
- USDT on 107 chains. USDC on 125. 214 distinct stablecoin assets.
- BIS (Shin, March 2026) proved this fragmentation is an equilibrium, not a magnusrary bug.
- $4.3B lost to bridge exploits trying to connect these chains.
- $7.2T monthly stablecoin volume (Feb 2026), overtaking ACH.

**What exists today and why it fails for retail users:**
- Bridges (Wormhole, LayerZero): require gas tokens on source chain. Can't help gasless users.
- BVNK/Bridge/Stripe: enterprise-only, $500K/month minimums.
- Arc/Magnus/Codex: institutional, no fiat rails, still require gas tokens per chain.
- Stellar: has anchors but no EVM, no compliance, no netting, 3-5s finality.

The gap: **nobody gives retail users a stablecoin wallet where deposits, sends, and withdrawals work without ever touching a gas token.** Where the fee is always in stablecoins. Where one balance routes to 125 chains and 195 countries. That's Magnus.

## Status Quo

Payment operators in emerging markets currently:
1. Pick one chain (usually Tron for USDT) and build around it
2. Handle compliance manually (or not at all)
3. Use centralized exchanges for FX conversion (Binance P2P, local OTC desks)
4. Connect to local fiat rails through custom integrations per country
5. No netting, every payment settles individually, high liquidity requirements

The result: fragmented, non-interoperable, compliance-questionable, expensive. Each operator builds their own silo. No network effects.

## Premises

1. Stablecoin fragmentation (125+ chains, 214+ tokens) is the #1 barrier to retail adoption. BIS, IMF, McKinsey, SoK paper all confirm this.
2. Bridges remain a structural vulnerability ($140M+ exploited in 2025-2026). Even improved bridges don't provide compliance, netting, or fiat rail connectivity. A hub-and-spoke settlement layer is structurally superior for payments.
3. Arc solves chain-to-chain but not chain-to-fiat-rail. The real gap is connecting stablecoins to local payment systems in emerging markets.
4. The fiat gateway protocol is the primary moat. Every fiat rail integration compounds the network effect, making it progressively harder for competitors to replicate.
5. Emerging market payment operators are the initial target. They can't access BVNK/Bridge and Arc doesn't serve their USDT-dominant, fiat-rail-dependent needs.
6. This is different from Magnus. Magnus was one chain for one market. This is a hub chain for all markets. Vietnam is the wedge, not the product.

### Core Components

Magnus is built on magnus-chain (a Magnus fork). Every stablechain (Arc, Magnus, Codex)
uses the same pattern: EVM + precompiles. Magnus follows this proven architecture and
adds the Gateway Protocol, bridges, and netting that no other stablechain has.

**1. Commonware Simplex Consensus (from Magnus, unchanged)**
- 200ms blocks, 300ms deterministic finality
- Threshold BLS certificates (~240 bytes) for cross-chain finality proofs
- No public mempool (FCFS ordering, no front-running)
- Same consensus used by Magnus ($5B, Visa as validator)

**2. Reth EVM + MIP20 Precompiles (from Magnus fork, unchanged)**
- MIP20 tokens: ERC-20 compatible precompiles with memo fields, ISO 20022 payment data, currency identifiers, compliance policies (MIP403). Same interface as Magnus's TIP-20.
- StablecoinDEX: enshrined order book precompile with limit orders, flip orders (perpetual liquidity), swaps. End-of-block matching. Same as Magnus's enshrined DEX.
- MipFeeManager: multi-stablecoin gas fees. Users pay in any registered stablecoin.
- Full EVM compatibility: Solidity, MetaMask, ethers.js, all Ethereum tooling works.

**3. Magnus Gateway Protocol — MGP (NEW, the moat)**
- Gateway precompile: RegisterGateway, GatewayDeposit, GatewayWithdraw, GatewaySettle
- Gateways are licensed entities registered on-chain with supported fiat rails
- Protocol-enforced escrow: user's stablecoins locked until gateway confirms fiat delivery
- Settlement attestation on-chain: auditable, slashable (Phase 1+)
- Like Stellar's SEP-6/24/31 but native to the chain, not application-layer
- Any wallet can pay any merchant on any Gateway via standard EVM calls

**4. Magnus Bridge Standard — MBS (Phase 2, NEW)**
- MagnusBridge.sol: one Solidity template (~500 LOC), deploy on any EVM chain
- Hub-and-spoke model: N connections (one per chain), not N-squared
- lockWithPermit(): gas-free deposits via ERC-2612 permit + relayer
- release(): BLS threshold signature verification (~48 bytes)
- lockAndRoute(): one-click cross-chain-to-fiat (lock + swap + gateway in one tx)
- Verified by Magnus validator set (same security as the chain)
- @magnus/bridge-sdk (npm) for app developers

**5. Multilateral Netting Engine (Phase 3, NEW)**
- Every N blocks (~20 seconds), compute net positions across all participants
- Target: 90%+ netting efficiency (CLS Bank achieves 96%)
- Reduces actual bridge usage by 90%+ (less bridge = less security risk)
- Enables zero destination gas fees (protocol absorbs gas from netting savings)
- No other L1 has protocol-level netting

**6. Compliance Pipeline (from Magnus fork, extended)**
- MIP403 policy registry: whitelist/blacklist per token, per jurisdiction (exists)
- Circuit breaker: emergency freeze precompile (Phase 3, new)
- Sanctions screening: precompile check on every transfer (Phase 3, new)
- Transfer limits per jurisdiction (Phase 3, new)
- Travel Rule: counterparty data required above threshold (Phase 3, new)
- Rules are data (governance transactions), not code

### Why not Stellar?

Stellar is the closest competitor and the protocol we most resemble architecturally (PathPayment, anchors, hub-and-spoke, emerging markets). Three reasons Magnus exists instead of building on Stellar:

1. **Anchors are application-layer, not protocol-native.** Stellar's SEP-6/24/31 standards define how anchors should behave, but the chain itself doesn't enforce gateway operations. A Stellar anchor can go offline, misreport reserves, or fail to honor withdrawals, and the protocol has no visibility. Magnus makes Gateway operations chain-native: registration, deposits, withdrawals, and settlements are all on-chain transactions with protocol-level guarantees.

2. **No EVM composability.** Stellar has its own VM (Soroban smart contracts). The $200B+ DeFi ecosystem runs on EVM. Magnus runs revm, so every registered stablecoin is automatically an ERC-20 accessible to existing Solidity tooling, wallets, and protocols. Stellar's ecosystem is isolated.

3. **No protocol-level compliance or netting.** Stellar has no built-in compliance pipeline (whitelists, sanctions screening, transfer limits, circuit breaker) and no multilateral netting. These are the features regulated institutions need and the features that differentiate Magnus from every chain in the market, including Stellar.

Stellar proved the model works. We're building the next-generation version. Stellar is SMTP. We're building Gmail.

### Zero Destination Gas Fees — How It Works

Users pay one fee in stablecoins (multi-currency gas) on Magnus. They never need ETH, SOL, TRX, or any other native gas token. Magnus handles destination chain gas internally.

```
USER: "Send 100 USDT to 0xABC on Ethereum"

1. Payment Engine debits 100 USDT + 0.05 USDT fee from user
2. Netting check: is there a pending inbound of ~100 USDT from Ethereum?
   YES → net it out, no bridge needed, instant settlement
   NO  → queue for bridge settlement
3. If bridge needed:
   - Burn rUSDT on Magnus
   - Produce finalization certificate (BLS threshold, ~240 bytes)
   - Relay certificate to Ethereum bridge contract
   - Bridge contract releases 100 USDT to 0xABC on Ethereum
   - Magnus pays ETH gas from protocol treasury
```

The netting engine is what makes the economics work. If 90% of transfers net out (CLS Bank achieves 96%), the protocol only pays destination gas for 10% of volume. The stablecoin fee on 100% of volume covers the gas cost on 10% of volume, plus profit.

For fiat rails, the same pattern: user pays stablecoin fee, Gateway handles the local fiat push. No gas tokens, no fiat fees visible to the user beyond the one settlement fee.

### Token Model Reconciliation (flow-through vs. DeFi)

**The reviewer caught a real contradiction.** Two token modes coexist:

1. **Routing tokens (transient):** rUSDC, rUSDT exist only during cross-chain settlement. Minted when source chain lock is verified, burned when destination chain release completes. These are the "flow-through" tokens.

2. **Gateway tokens (persistent):** gVND, gUSD issued by Gateways represent fiat deposits. These persist on-chain because they represent actual fiat held by Gateways. DeFi can interact with these.

3. **Issuer tokens (persistent):** If Circle/Tether deploy natively (Phase 3+), their tokens persist on-chain like any ERC-20. DeFi composable.

Routing tokens are transient. Gateway and issuer tokens are persistent. DeFi operates on persistent tokens. No contradiction.
